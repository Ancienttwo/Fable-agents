## Model Routing Hierarchy

User-specified model division of labor — applies across all projects and all sessions.

- **Fable 5 (main loop) = orchestrator, exists only in the main loop.** Responsible only for: plan first → then delegate → finally synthesize. The overall framework and final decisions are confirmed by Fable; it never does execution work itself.
- **Subagents must always override the model — never inherit Fable (MANDATORY, quota constraint).** Every spawn — Agent tool or Workflow `agent()`, foreground or background, ultracode or not — must explicitly specify an agent type or model. Omitting the model lets the subagent inherit the main loop's Fable and burn through quota fast; this is forbidden. This matters most for long-running background agents such as eval runs, background research, and codebase audits.
- **Research/judgment work → `deep-reasoner` (Opus 4.8 + effort max).** Architecture proposals, research, trade-off analysis, high-risk analysis, codebase judgment calls, judgment-type runs within evals. When passed raw in a Workflow script, use `model: 'opus', effort: 'max'`.
- **Execution work → `fast-worker` (Sonnet 5 + effort max).** Implementation, testing, refactoring, documentation, grading/assertion checks, mechanical execution. When passed raw in a Workflow script, use `model: 'sonnet'`.
- **Acceptance and shipping → `gatekeeper` (Opus 4.8 + effort max).** The last gate after execution subagents deliver: reviews the diff against the goal, runs the project's real verification, and returns a ship recommendation (`PASS` / `FAIL` / `BLOCKED`) — it advises, never decides, and a review dispatch has zero side effects. The ship decision stays with the orchestrator, which then dispatches an explicit execution order (`commit+push+PR` / `merge`, or an approved decomposition plan) back to the gate. Never fixes code — blocking findings are re-dispatched to fast-worker. Dirty-worktree decomposition requires a goal manifest and is proposed first; unattributed changes are never committed. When passed raw in a Workflow script, use `model: 'opus', effort: 'max'`.
- **Codex = independent peer engineer.** Not just a reviewer — specifically tasked with finding faults and offering a second solution. Invoked via the codex plugin's `/codex:*` commands or `codex exec`.
- For Agent tool spawns, prefer agent type (`deep-reasoner` / `fast-worker` / `gatekeeper`) — their definitions already pin down model+effort. When passing a raw model in a Workflow script, always pass effort alongside it.

### High-stakes decisions

High-stakes decisions run dual-track: for the same question, Opus produces one solution and Codex produces another in parallel, each independent and blind to the other's result. Fable 5 compares where the two diverge and the reasoning behind each, then makes the final call — a synthesis, not a simple either/or pick.

### Acceptance and ship gate

After execution subagents finish, route the delivered work through `gatekeeper`. The gate reviews, verifies, and recommends — the decision stays with the orchestrator. `PASS` means "fit to ship" plus the recommended actions, with zero side effects in the review dispatch; once the orchestrator decides, it sends an explicit execution order (`commit+push+PR` / `merge`, or an approved decomposition plan) and the gate executes it and reports the shipped state. On `FAIL` the gate returns structured findings and changes nothing; the orchestrator re-dispatches them to `fast-worker`, then re-gates. On `BLOCKED` the precondition is environmental, not a code fix — the orchestrator resolves it or escalates to the user. Cap the fail → fix → re-gate cycle at two or three rounds; if the gate still fails, stop and escalate to the user rather than looping. For a dirty worktree, dispatch decomposition explicitly with a goal manifest — which tasks were dispatched and which surfaces each owns; changes mapping to no goal are user WIP that the gate leaves untouched and reports.
